## Linux PAM Authentication: How Modules Are Invoked During sudo and SSH

### Introduction

Linux applications such as `sudo`, `sshd`, `login`, and `su` need a way to authenticate users. Rather than implementing authentication themselves, they rely on the **Pluggable Authentication Modules (PAM)** framework.

This article explains:

+ What PAM is
+ How PAM modules are invoked
+ How sudo authentication works
+ How SSH authentication works
+ A hands-on lab to observe PAM in action

### What is PAM?

PAM provides a standardized authentication framework for Linux applications.

Without PAM:

```
Application
 ├─ Password verification
 ├─ Account checks
 ├─ Session management
 └─ Authorization logic
```

With PAM:

```
Application
      │
      ▼
 PAM Library
      │
      ▼
 PAM Configuration
      │
      ▼
 PAM Modules
```

Applications simply ask PAM:

> "Please authenticate this user."

PAM then determines which modules to execute and in what order.

### PAM Components

#### PAM Configuration Files

Located under `/etc/pam.d` directory.

Examples:

```
/etc/pam.d/sudo
/etc/pam.d/sshd
/etc/pam.d/login
/etc/pam.d/su
```

Each file defines the PAM stack for a specific service.

#### PAM Modules

Modules are shared libraries typically found in:

```
/usr/lib64/security/
```
or

Examples:
```
pam_unix.so
pam_limits.so
pam_env.so
pam_nologin.so
pam_exec.so
pam_warn.so
```

#### Understanding PAM Rule Syntax

A PAM rule consists of:
```
type    control    module-path    arguments
```

Example:
```
auth required pam_unix.so
```

#### PAM Types

| Type | Purpose | 
  |------|-------|
  | auth | Authentication | 
  | account | Account validation |
  | password | Password changes |
  | session | Session setup/cleanup |

#### Control Flags

| Type | Purpose | 
  |------|-------|
  | required | Must succeed | 
  | requisite | Must succeed immediately |
  | sufficient | Success may short-circuit processing |
  | optional | Usually ignored |

### Example 1: PAM During sudo Authentication

#### Step 1: User Runs sudo

```
sudo ls
```

Execution flow:
```
User
 │
 ▼
sudo
 │
 ▼
PAM Library
 │
 ▼
/etc/pam.d/sudo
 │
 ▼
Referenced PAM Modules
```

#### Step 2: Inspect sudo PAM Configuration

```
cat /etc/pam.d/sudo
```

Example:
```
@include common-auth
@include common-account
@include common-session-noninteractive
```

#### Step 3: Authentication Module Execution

Inspect:
```
cat /etc/pam.d/common-auth
```

Example:
```
auth required pam_unix.so
```

Execution:
```
sudo
 │
 ▼
pam_unix.so
 │
 ▼
Verify password in /etc/shadow
 │
 ▼
Success or Failure
```

#### sudo PAM Flow Diagram

```
sudo ls
   │
   ▼
sudo invokes PAM
   │
   ▼
common-auth
   │
   ▼
pam_unix.so
   │
   ▼
Password validated
   │
   ▼
common-account
   │
   ▼
common-session
   │
   ▼
Command executes
```

### Example 2: PAM During SSH Authentication

When a user connects:
```
ssh user@server
```

The SSH daemon (`sshd`) delegates authentication to PAM.

#### Inspect SSH PAM Configuration

```
cat /etc/pam.d/sshd
```

Example:
```
@include common-auth
account required pam_nologin.so
@include common-account
session required pam_limits.so
@include common-session
```

#### SSH Authentication Flow
```
ssh client
      │
      ▼
sshd
      │
      ▼
PAM
      │
      ▼
common-auth
      │
      ▼
pam_unix.so
      │
      ▼
Password verified
      │
      ▼
pam_nologin.so
      │
      ▼
Check login restrictions
      │
      ▼
pam_limits.so
      │
      ▼
Apply user limits
      │
      ▼
User shell started
```

### Hands-On Lab: Observe PAM Module Execution

#### Lab Objective

Use PAM logging modules to see when PAM is invoked.

#### Prerequisites
+ Test VM
+ Root access
+ Active SSH session kept open

> Never test PAM changes on a production server.

#### Step 1: Add Logging Module to sudo
Backup first:
```
sudo cp /etc/pam.d/sudo /etc/pam.d/sudo.bak
```

Edit:
```
sudo vi /etc/pam.d/sudo
```

Add near the top:
```
auth optional pam_warn.so
```

Example:
```
auth optional pam_warn.so
@include common-auth
@include common-account
@include common-session-noninteractive
```

#### Step 2: Trigger PAM
Run:
```
sudo ls
```

#### Step 3: View Logs
On systems using systemd:
```
sudo journalctl -xe | grep PAM
```

You should see messages generated by `pam_warn.so`.

This proves:
```
sudo
  │
  ▼
PAM
  │
  ▼
pam_warn.so executed
```

### Hands-On Lab: Execute a Custom Script Using PAM

One of the most powerful PAM modules is:
```
pam_exec.so
```

It executes a custom script whenever PAM reaches that rule.

#### Step 1: Create a Script
```
sudo vi /usr/local/bin/pam-test.sh
```

Contents:
```
#!/bin/bash

echo "$(date) PAM invoked by $PAM_SERVICE for $PAM_USER" >> /tmp/pam.log
```

Make executable:
```
sudo chmod +x /usr/local/bin/pam-test.sh
```

#### Step 2: Add PAM Rule
Edit:
```
sudo vi /etc/pam.d/sudo
```

Add:
```
auth optional pam_exec.so /usr/local/bin/pam-test.sh
```

Example:
```
auth optional pam_exec.so /usr/local/bin/pam-test.sh
@include common-auth
@include common-account
@include common-session-noninteractive
```

#### Step 3: Trigger Authentication
Run:
```
sudo ls
```

#### Step 4: Verify Script Execution
```
cat /tmp/pam.log
```

Example output:
```
Thu Jun 2 09:30:15 UTC 2026 PAM invoked by sudo for faizyakob
```

This proves a PAM module was executed during authentication.


### Lab: Break SSH Login Using PAM (Safely)

⚠️ Perform this only in a test VM.

Keep an existing root session open.

#### Step 1: Enable Login Restriction
Create:
```
sudo touch /etc/nologin
```

#### Step 2: Verify PAM Configuration
```
grep pam_nologin /etc/pam.d/sshd
```

#### Step 3: Attempt SSH Login
```
ssh user@server
```

Expected:
```
System is going down for maintenance.
```

or

```
Login denied.
```

#### Step 4: Restore Access
Remove:
```
sudo rm /etc/nologin
```

SSH logins should work again.

### Key Takeaways

+ Applications such as `sudo` and `sshd` do not authenticate users directly.
+ They invoke PAM through the PAM library.
+ PAM reads service-specific configuration from `/etc/pam.d/`.
+ Modules are executed sequentially according to the PAM stack.
+ `pam_unix.so` performs traditional password authentication.
+ `pam_warn.so` can be used to observe PAM activity.
+ `pam_exec.so` can execute custom scripts during authentication.
+ Misconfigured PAM rules can lock users out of a system, so always test carefully.
